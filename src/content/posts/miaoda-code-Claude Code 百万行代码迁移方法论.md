# 用 Claude Code 完成百万行代码迁移？Anthropic 官方方法论详解

## 背景

Anthropic 近期发布了一篇官方博文，揭秘了他们内部如何利用 Claude Code 完成大规模代码迁移的实战经验。文中两个案例堪称震撼：

- **Bun 项目**：创始人 Jarred Sumner 用 Claude Code 将 Bun 从 Zig 语言重写为 Rust，**53 万行代码仅用 11 天**完成，100% 测试通过后合并上线。
- **Python 转 TypeScript**：Anthropic Labs 联合负责人 Mike Krieger 花一个周末，将一个 Python 项目迁移为 **16.5 万行 TypeScript**，构建时间从 30 分钟骤降至约 2 秒。

这种量级的工程迁移，传统方式往往需要顶尖团队耗费一两年，且难以保证完全兼容。而现在，一个人配合 AI 就能在几天内完成。

本文结合官方博文、Bun 技术博客以及实战经验，对这套方法论进行完整的中文深度解读。

---

## 核心思路：从"改代码"到"设流程"

传统代码迁移是逐文件翻译、逐个检查、逐个修 bug——几十文件规模尚可，成百上千文件就不可行了。

**AI 时代的正确思路是：把注意力放在产出代码的流程上，而不是代码本身。**

当发现 AI 在某类翻译上反复出错时，不要逐个修复代码，而是去**修复规则手册**，让后续所有翻译都不再犯同样的错。规则越完善，人工干预就越少。

举个例子：如果 AI 写 Spring Boot 接口总忘记加参数校验注解，就在规则文档里加一条：

> 所有 Controller 层请求参数必须使用 `@Valid` + DTO 校验，禁止在 Service 层手动 if 判空。

此后这个问题就彻底消失了。

**核心原则：出了问题不要只盯着问题本身，要修复产出问题的源头。**

## 六步迁移法

Anthropic 总结了一套完整的大规模代码迁移流程，共六个步骤。

### 第 0 步：建立验证机制

**迁移之前必须先搞定验证手段，否则你永远不知道什么时候算完成。**

最理想的情况是已有完整的测试套件，且测试用第三方语言编写（如 Bun 的测试用 TypeScript，不依赖 Zig 本身）。

如果测试与源代码同语言：
- 区分**外部接口测试**（可直接复用）和**内部实现测试**（需重写为语言无关形式）

如果没有现成测试：
- 编写**对比脚本**：跑若干真实场景，对新旧版本输出做 diff，任何行为差异都算 Bug
- 小项目也适用：准备几组真实输入输出样本，迁移后逐一比对

### 第 1 步：制定规则手册与依赖图

这是人工投入最多的阶段。

Jarred（Bun 创始人）仅与 Claude 讨论 Zig 到 Rust 的模式映射就花了 3 小时，最终产出一份 576 行的规则手册，详细规定了每种类型映射、每种惯用法转换。

**规则手册的内容取决于核心策略选择：**

| 策略 | 规则手册性质 | 适用场景 |
|------|-------------|---------|
| 保持原有架构逐行翻译 | 类型/惯用法对照表 | Bun 案例（机械翻译） |
| 完全重新设计架构 | 新系统设计文档 | Mike 的 Python→TS 案例 |

**依赖图**同样关键：明确文件间依赖关系，才能决定迁移顺序和批次。对于 Python 这类隐式依赖语言，可让 AI 写脚本自动分析。

此外还要做一份**差异清单**，列出两种语言之间无法简单翻译的核心差异（如 Zig 手动内存管理 vs Rust 所有权、Python 动态类型 vs TypeScript 显式接口）——这些是 AI 最容易出错的地方，必须在规则中重点标注。

### 第 2 步：小范围试跑

规则写好后别急着全量跑，**先拿 3 个文件试水**。

Jarred 的做法非常精妙：
- 一个 Claude 实例按规则手册翻译
- 另一个 Claude 以"高级 Rust 工程师"身份独立翻译
- 第三个全新 Claude 专门对比两个版本，从差异中提取新规则

这一步他发现了 2 个关键问题——如果直接铺开 1448 个文件，后果不堪设想。

对于重新设计架构的项目，Mike 的做法是让多个 Claude 从不同角度挑设计文档的毛病，然后跑完整端到端翻译验证设计可行性。**他实际上跑了三次完整迁移，前两次全部丢弃产出，只保留规则改进，第三次才正式保留结果。**

> 启示：不要追求一步到位。先用粗糙版本完善规则和提示词，磨刀不误砍柴工。

### 第 3 步：全量翻译

规则经试跑验证后，进入全量执行阶段。

**核心架构是三角色流水线：**

```
翻译 Agent → 找茬 Agent ×2 → 修复 Agent
```

- **翻译 Agent**：按规则手册批量翻译文件
- **找茬 Agent**（对抗性审查）：两个独立实例专门挑毛病，心态是"假设这段代码有 Bug"
- **修复 Agent**：处理审查发现的问题

**三个实操细节：**

1. **工作队列机械化**：通过检查磁盘上是否存在目标文件判断进度，流程天然可恢复，中断后重启即可
2. **翻译与审查对话隔离**：审查者必须在全新对话中工作，只看翻译结果，不看翻译推理过程——写代码的 AI 总倾向认为自己没问题
3. **模型分层使用**：翻译这种高并发工作可用较便宜模型（如 Claude Sonnet），规则制定和审查用最强模型（如 Claude Opus）。Mike 在全量阶段就用了 12 个 Sonnet 子代理并行

拿不准的地方直接标 `// TODO(port): 原因`，不要在翻译阶段纠结，编译器和测试后续会给出答案。

### 第 4 ~ 6 步：编译 → 运行 → 行为对齐

后三步套路一致：**跑一遍 → 收集错误清单 → 批量修复 Agent 并行处理 → 再跑一遍 → 循环至清零**。人工介入程度逐级降低。

**第 4 步 · 编译阶段**
- 对整个工作区跑编译器，错误按模块分组输出
- 64 个修复 Agent 并行处理，每个配 2 个对抗性审查者
- Jarred 遇到的大坑：Zig 单包编译拆成 Rust 100 个模块后引入大量循环依赖，修复后暴露约 16000 个编译错误——64 个并行 Claude 几小时搞定

**第 5 步 · 冒烟测试阶段**
- 各子命令逐一运行，收集崩溃信息和报错栈
- 同样的"修复 + 审查"循环处理

**第 6 步 · 行为对齐阶段**
- 测试套件分片运行，每个失败测试交给一个修复 Agent
- 修复后由对抗性审查者检查

Jarred 还有一个巧妙设计：**专门的构建守护进程统一编译**，修复 Agent 只提交代码，守护进程定期批量编译、跑受影响测试、反馈结果——避免多 Agent 各自触发编译造成资源冲突和重复。

Bun 的 CI 从 972 个测试文件失败到全部通过，耗时约 4 天，Linux 最先通过，Windows 最后。合并后共出现 19 个回归 Bug，均已修复。

> 类似案例：Airbnb 用 AI 将 3500 个 React 组件测试从 Enzyme 迁移到 React Testing Library，原估一年半，最终 6 周完成。

## 关键洞察与实践建议

### 对抗性审查值得日常养成
写代码的 AI 和审代码的 AI 一定要分开。Claude Code 中有内置 `/code-review` 命令，也可以开新对话做独立审查——哪怕不做代码迁移，这个习惯也能显著提升代码质量。

### 成本账要算明白
Bun 整个迁移花费了 **16.5 万美元 API 费用**。听起来惊人，但对比 3 名高级工程师一年的薪资加机会成本，已经非常划算。

对个人开发者而言，几万行项目的迁移通常 Pro/Max 订阅额度就够用了。值不值，用自己的人力成本折算一下就清楚。

### 不要盲目迁移
现有代码跑得稳、维护成本不高，就没必要为了迁移而迁移。**迁移的前提是存在持续的真实痛点。**

### 人的判断力不可替代
Jarred 在整个迁移过程中，每天都在监控工作流输出、手动检查 AI 行为、发现系统性问题后调整流程。AI 执行力很强，但决策权始终在人手上——AI 能省成本，不代表完全不需要人。

---

## 即刻可用的四条方法论总结

1. **先定规则再动手**：稍大的重构或迁移，先和 AI 聊出一份规则文档，把"什么该怎么改"定义清楚
2. **小范围试跑补规则**：先拿 3 个文件试水，发现的问题不要逐个修代码，补进规则文档后重新生成
3. **写和审必须分离**：用独立对话或专门的 code review 工具做对抗性审查
4. **错误即任务清单**：编译报错、测试失败、Lint 警告都是天然的待办列表，让 AI 按清单逐项修复

几十万行、上百万行的代码迁移 AI 都能完成，日常几千几万行的项目重构更不在话下。**关键不在于 AI 有多聪明，而在于你给 AI 设计的流程够不够好。**
